# Polymarket 예측·리스크 모듈 설계 명세서 (for Claude)

## 0. 시스템 개요

목표:
폴리마켓에서 “모델이 추정한 실제 발생 확률”과 “시장 가격이 내포하는 확률”의 차이를 이용해 장기적으로 양(+)의 기대 수익을 갖는 자동 매매 시스템을 설계·구현한다.

핵심 개념:

- 각 이진(or 다중) 시장에 대해:
    - $P_M$: 시장 암묵 확률 (가격에서 유도)
    - $P_R$: 모델 추정 실제 확률
    - $\alpha = P_R - P_M$: 알파(오차) 시그널
- 트레이딩 로직은 $\alpha$를 입력으로 포지션 방향과 크기를 결정한다.
- 별도의 리스크 모듈이 포지션·포트폴리오 한도를 강제한다.

Claude의 역할:

- 외부 API/데이터 소스를 호출하는 툴을 사용해 데이터 수집·주문 실행.
- 텍스트 데이터(이벤트 설명, 뉴스, 트윗 등)를 요약·스코어링하여 피처로 구조화.
- 예측/리스크 규칙에 따라 일관된 의사결정 수행.

***

## 1. 전체 아키텍처

모듈 구조(논리적):

1. 데이터 모듈 (Data Layer)
2. 예측 모듈 (Prediction Layer)
3. 시그널·실행 모듈 (Signal \& Execution Layer)
4. 리스크·포트폴리오 모듈 (Risk \& Portfolio Layer)
5. 모니터링·피드백 모듈 (Monitoring \& Feedback)

각 모듈은 함수/툴 호출 단위로 분리 가능한 형태로 설계한다.

## 2. 데이터 모듈 명세

### 2.1. 입력 데이터 종류

Claude는 다음 데이터를 주기적으로 또는 이벤트 기반으로 수집해야 한다.

1) 폴리마켓 온체인/오프체인 데이터

- 마켓 리스트:
    - 시장 ID, 제목, 설명, 카테고리(정치/크립토/스포츠 등), 마감 시간, 상태(open/closed/settled)
- 시세 데이터:
    - 각 마켓·옵션(예: YES/NO)에 대한 시간별 가격
    - 스냅샷 기준: 최소 1분~5분 단위, 설정 가능
- 오더북 데이터:
    - 각 옵션의 bid/ask 가격 및 잔량 상위 N레벨 (예: 10 레벨)
- 체결(트레이드) 데이터:
    - 체결 가격, 수량, timestamp

2) 외부 정보 데이터

- 뉴스/헤드라인 텍스트
- 트위터/소셜 미디어 텍스트
- 여론조사/설문 결과(정치·사회 이슈일 경우)
- 온체인 데이터(필요 시, 특정 크립토 관련 이벤트)


### 2.2. 데이터 모듈이 제공해야 할 기능(인터페이스 수준)

Claude는 다음과 같은 “기능”을 사용할 수 있어야 한다 (실제 구현은 별도 툴로 제공된다고 가정).

- `get_markets(filters) -> list[Market]`
    - 필터: 상태(open), 카테고리, 마감 시간 범위 등
- `get_market_history(market_id, start, end, interval) -> OHLCV-like series`
- `get_orderbook(market_id) -> orderbook_snapshot`
- `get_trades(market_id, start, end) -> list[Trade]`
- 외부 데이터 관련:
    - `get_news(query, since) -> list[Article]`
    - `get_tweets(query, since) -> list[Tweet]`
    - `get_polls(event_key) -> list[Poll]` (선택적)

데이터는 가능한 한 구조화된 형태(JSON 등)로 반환되며, Claude는 이를 바로 피처 엔지니어링에 사용할 수 있어야 한다.

## 3. 예측 모듈 명세

예측 모듈의 목적: 각 마켓에 대해 “실제 발생 확률” $P_R$을 추정하고, 시장 암묵 확률 $P_M$와의 차이 $\alpha$를 계산해 시그널을 생성한다.

### 3.1. 시장 암묵 확률 $P_M$

- 이진 시장의 YES 토큰 가격을 $p$라 할 때, 수수료·구조가 단순하다고 가정하면:
    - $P_M = p$ (0~1 사이 값)
- 필요 시 수수료·리베이트 구조를 반영한 보정 로직을 별도 함수로 둔다.
- 다중 선택 시장의 경우, 각 옵션 가격을 정규화해 확률로 변환한다.

Claude가 해야 할 일:

- 현재 가격으로부터 각 옵션의 $P_M$ 계산
- 계산 결과를 예측 모듈의 내부 데이터 구조에 저장


### 3.2. 모델 추정 확률 $P_R$

Claude는 하나 이상의 모델을 사용해 $P_R$을 추정해야 한다. 모델은 크게 세 가지 요소로 구성된다.

1) 구조화 피처(수치형/범주형)
2) 텍스트 기반 피처(LLM 기반 스코어)
3) 앙상블 논리

#### 3.2.1. 구조화 피처 예시

각 마켓에 대해, 최소 다음 피처 세트를 정의한다.

- 마켓 메타 정보:
    - 카테고리(정치/크립토/스포츠/기타)
    - 마감까지 남은 시간(초/분/시간 단위)
- 가격/유동성 관련:
    - 최근 1h/24h 수익률
    - 최근 1h/24h 변동성(표준편차)
    - 24h 거래량
    - 현재 스프레드(최우선 ask - bid)
    - 오더북 비대칭도(상위 N레벨 bid vs ask 잔량 비율)
- 과거 패턴 관련:
    - 유사 카테고리·구조(예: “대선 승리 여부”) 이벤트의 과거 데이터에서 추출한 통계값(선택적)

Claude는 위와 같은 피처를 생성하는 함수를 설계해야 한다.

예:

- `build_structured_features(market_id, now_timestamp) -> FeatureVector`


#### 3.2.2. 텍스트 기반 피처 (Claude 사용)

Claude는 이벤트 설명, 관련 뉴스, 트윗 등을 입력으로 받아 텍스트 기반 피처를 생성한다.

- 입력:
    - 마켓 제목, 설명
    - 최근 관련 기사 제목/요약
    - 최근 트윗 내용(핵심만 샘플링)
- 출력(예시):
    - 감성 점수 (−1 ~ 1 범위)
    - 불확실성/컨센서스 정도(0 ~ 1)
    - 주요 리스크 요인 카운트(부정적 키워드 수 등)
    - 한 줄 요약(모니터링용)

Claude는 자체적으로:

- “이 이벤트가 YES로 끝날 가능성이 얼마나 높다고 느껴지는지”에 대한 직관적 점수를 생성할 수도 있으나, 이 값은 구조화 모델의 보조 피처로 사용한다(직접 확률로 사용하지 않고, 피처로 주입).

인터페이스 예:

- `build_text_features(market_id, related_texts) -> TextFeatureVector`


#### 3.2.3. 예측 모델 구성

Claude는 최소 두 종류의 모델을 가정한다.

1) 베이스라인 로지스틱 회귀(Logistic Regression)
2) 비선형 트리 기반 모델(XGBoost/LightGBM 등) — 개념적으로만 정의

학습 방식:

- 학습 데이터:
    - 과거 마켓들에 대해, 마감 시점의 실제 결과(YES=1, NO=0)와
    - 마감 N시간 전 시점의 피처 벡터를 매칭
- 학습 주기:
    - 롤링 윈도우: 최근 60~120일 데이터를 사용해 주기적으로 재학습
- 출력:
    - 각 모델별 $P_R^{(model)}$

앙상블:

- 여러 모델의 $P_R^{(model)}$를 가중 평균하여 최종 $P_R$ 산출
- 카테고리별로 다른 가중치를 사용할 수 있도록 설계
- 고수준 인터페이스:
    - `predict_probability(feature_vector) -> P_R (0~1 float)`

Claude는 구체적인 수치 최적화나 라이브러리 호출 대신, 위 함수가 존재한다고 가정하고 이를 사용하는 방향으로 설계를 진행한다.

### 3.3. 알파 계산 및 시그널

- $\alpha = P_R - P_M$
- 임계값(Threshold) $\tau_\alpha$를 둔다.
    - $|\alpha| \le \tau_\alpha$ → 노트레이드
    - $\alpha > \tau_\alpha$ → YES 롱(또는 NO 숏 축소)
    - $\alpha < -\tau_\alpha$ → NO 롱(또는 YES 숏)

Claude는 다음 함수를 사용한다고 가정한다.

- `compute_alpha(P_R, P_M) -> alpha`
- `generate_signals(alpha, market_state, risk_limits) -> TradeSignal`

## 4. 시그널·실행 모듈 명세

### 4.1. 포지션 방향 결정

입력:

- $\alpha$
- 시장 상태(마감까지 남은 시간, 유동성 상태 등)
- 현재 보유 포지션

기본 규칙:

- $\alpha > \tau_\alpha$: YES 매수/롱
- $\alpha < -\tau_\alpha$: NO 매수/롱
- 이미 반대 방향 포지션 보유 시:
    - 우선 기존 포지션 축소 또는 청산 후 신규 포지션 진입


### 4.2. 포지션 크기 결정(리스크 모듈과 연동)

포지션 사이징은 5장 리스크 모듈의 규칙을 따른다. 이 모듈에서는 “방향 신호”와 “기본 강도”만 결정한다고 가정한다.

예:

- `base_size_factor = f(|alpha|)` 형태 (예: $|\alpha|$에 비례)

실제 주문 크기 결정은 리스크 모듈에서 전체 포트폴리오 한도 등을 고려해 최종 수치를 산출한다.

### 4.3. 주문 생성·관리

Claude가 사용할 수 있어야 할 기능(개념적):

- `place_order(market_id, side, size, price, type) -> order_id`
- `cancel_order(order_id)`
- `get_open_positions() -> list[Position]`
- `close_position(position_id, mode)`

실행 모듈의 책임:

- 슬리피지 허용 범위 내에서만 주문을 제출
- 지정가/시장가 선택 로직
- 체결 여부 모니터링 및 재호가 재조정 로직

## 5. 리스크·포트폴리오 모듈 명세

리스크 모듈은 **항상 최종 승인 권한**을 가진다. 어떤 시그널이 나오더라도 리스크 한도를 위반하면 주문을 차단해야 한다.

### 5.1. 포트폴리오 한도

설정 가능한 한도(설정 파일/YAML 등에서 읽어들인다고 가정):

- 단일 이벤트 최대 익스포저 비율:
    - 예: 총 자본의 X% 이상을 동일 이벤트에 배팅 금지
- 상관관계 이벤트 묶음 한도:
    - 같은 테마(예: 같은 선거, 같은 자산에 대한 조건부 이벤트)끼리 그룹핑 후 그룹별 최대 비율 설정
- 카테고리별 비중 한도:
    - 정치, 크립토, 스포츠 등 카테고리별 최대 비중

Claude는 다음 평가 함수를 사용해야 한다.

- `check_exposure_limits(candidate_trade, current_portfolio, limits) -> allowed(bool), reason(str)`


### 5.2. 포지션 사이징 (Kelly 기반 응용)

개념:

- Kelly 공식을 기본 아이디어로 사용하되, 실제 적용은 fraction(부분 Kelly)만 사용
- 입력:
    - 승률 추정 $P_R$
    - 기대 승/패 비율(bet payoff 구조)
- 출력:
    - 권장 베팅 비율 $f$
- 실제 $f$는 0.25~0.5 정도로 스케일 다운할 것을 전제로 설계

Claude는 “정확한 수식 계산” 대신, 다음과 같은 고수준 로직을 따른다.

- $|\alpha|$가 클수록 베팅 비율 증가
- 전체 계좌 변동성을 제한하기 위해 상한값(cap)을 둔다

함수 예:

- `compute_position_size(alpha, capital, volatility_estimate, kelly_fraction_limits) -> size`


### 5.3. 슬리피지·유동성 제어

규칙:

- 슬리피지 캡:
    - 주문당 허용 가능한 미끄러짐 범위 설정(예: 0.5~1%p)
- 유동성 필터:
    - 최근 24h 거래량·현재 오더북 깊이가 기준 이하이면:
        - 주문 크기 축소 또는 진입 금지

Claude는 시그널을 주문으로 바꾸기 전에 다음을 수행해야 한다.

- `estimate_slippage(market_id, desired_size, side) -> estimated_slippage`
- `check_liquidity(market_id) -> liquidity_score`

슬리피지 및 유동성 조건을 만족하지 않으면:

- 시그널을 폐기 또는 크기 축소


### 5.4. 글로벌 리스크 관리

전 계좌 수준에서 다음 규칙들을 강제:

- 일간 손실 한도:
    - 하루 누적 손실이 계좌의 X% 초과 시:
        - 신규 진입 금지
        - 기존 포지션 점진적 축소
- 주간 손실 한도(선택적):
    - 유사한 방식으로 적용
- 이벤트 리스크:
    - 큰 이벤트(예: 주요 지표 발표, 대선 토론 등) 이전에 노출 축소
    - 특정 시각 전에 포지션 강제 축소 혹은 청산 규칙 설정

Claude는 주기적으로:

- `evaluate_portfolio_risk(current_portfolio, pnl_history, risk_limits) -> actions_to_take`

를 호출하는 형태로 로직을 설계해야 한다.

## 6. 모니터링·피드백 모듈

Claude는 전략 성과를 지속적으로 평가하고, 예측 모델 성능을 추적해야 한다.

### 6.1. 성과 지표

전략 관점:

- PnL (누적, 일간)
- Sharpe ratio (간단 추정)
- 최대 낙폭(MDD)
- Hit rate (수익 거래 비율)

예측 관점:

- Brier score (확률 예측 정확도)
- Log loss
- 예측 확률 구간별 실제 빈도(캘리브레이션 검토용)

Claude는 일정 주기(예: 하루 1회)로:

- 전략별 성과 요약 텍스트 생성
- “어떤 카테고리/이벤트 유형에서 알파가 잘/못 나오는지” 분석
- 필요시:
    - 임계값 $\tau_\alpha$, Kelly fraction, 카테고리별 가중치 등을 조정하는 제안 생성

## 7. Claude가 앞으로 구현해야 할 구체 작업 목록

Claude는 이 문서를 바탕으로 다음 단계별로 설계를 진행해야 한다.

1) **데이터 인터페이스 설계**
    - 마켓/시세/오더북/체결/외부 텍스트 데이터를 가져오는 함수·툴 시그니처를 정의
    - 최소 입력·출력 스키마를 명확히 문서화
2) **피처 엔지니어링 설계**
    - `build_structured_features`, `build_text_features`의 입력·출력 포맷 정의
    - 카테고리별(정치/크립토/스포츠) 필수 피처 항목 목록화
3) **예측 모듈 설계**
    - `predict_probability` 함수의 내부 논리(로지스틱 + 트리 모델 앙상블 개념) 정의
    - 학습/재학습 스케줄, 피처 저장 형식, 라벨링 규칙 설계
4) **알파·시그널 생성 로직 설계**
    - `compute_alpha`, `generate_signals`의 규칙과 파라미터(임계값 등) 명시
    - 노이즈를 줄이기 위한 필터링 조건 정의(예: 최소 유동성 조건)
5) **리스크·포트폴리오 모듈 설계**
    - 포지션/포트폴리오 한도 구조 정의 (이벤트·그룹·카테고리)
    - `check_exposure_limits`, `compute_position_size`, `evaluate_portfolio_risk`의 규칙 및 파라미터 정의
    - 손실 한도와 동작(트레이딩 중지, 디레버리징 등) 정의
6) **실행 모듈 및 주문 관리 설계**
    - 주문 생성/취소/체결 모니터링 인터페이스 정의
    - 슬리피지·유동성 검사 절차를 주문 전 필수 단계로 포함
7) **모니터링·피드백 루프 설계**
    - 성과 지표 계산 로직
    - 주기적 리포트 생성 포맷
    - 파라미터 자동/반자동 조정 제안 로직

이후 단계에서, 각 인터페이스(예: `get_markets`, `predict_probability`, `compute_position_size` 등)에 대해 구체적인 타입 정의와 의사코드를 추가하는 형태로 문서를 확장하면 된다.
